# Платёжная процедура (ручная оркестрация)

## Модель оплаты

Платформа использует **ручную оркестрацию платежей** через админа. Прямых интеграций с платёжными шлюзами нет (YooKassa удалена в апреле 2026).

**Способы оплаты для заказчика:**
- СБП (Система Быстрых Платежей) — перевод по номеру телефона / счёта
- Банковский счёт (для ИП/ООО)
- Наличные (Омск, при встрече)

**Способы выплаты исполнителю:**
- СБП (перевод на карту)
- Наличные
- Банковский перевод (по реквизитам)

## Пошаговая процедура

### 1. Заказчик оставляет заявку

1. Заказчик заполняет форму на лендинге (POST `/api/orders` или `/api/orders/create`)
2. Заказ создаётся в БД со статусом `pending`
3. Админ получает уведомление (MAX / Telegram)

### 2. Админ оценивает и выставляет счёт

1. Админ открывает заказ в админ-панели: `/admin` → Заказы
2. Оценивает стоимость (если не проставлена автоматически):
   - `display_price` — цена для заказчика (скрытая наценка уже включена)
   - `supplier_payout` — сумма выплаты исполнителю
3. Статус → `priced`
4. Админ нажимает «Отправить счёт» → статус → `payment_sent`
5. Система отправляет заказчику ссылку на dashboard через MAX/Telegram
6. Заказчик видит `display_price` на странице `/my/:token`

### 3. Заказчик оплачивает

1. Заказчик переводит сумму по СБП:
   - **Получатель:** ИП Подряд ПРО (или физлицо)
   - **Телефон/Счёт:** реквизиты из `.env` (`COMPANY_ACCOUNT`)
   - **Сумма:** `display_price` (видна в dashboard)
   - **Назначение:** «Заказ № [order_id]»
2. Заказчик присылает скриншот/чек админу через MAX/Telegram
3. **Важно:** не менять статус до фактического получения денег на счёте!

### 4. Админ подтверждает оплату

1. Проверить поступление в банковском приложении
2. Сверить сумму и назначение платежа с `order_id`
3. В админ-панели: найти заказ → «Подтвердить оплату»
4. Или через API: `PUT /api/admin/orders/:id/payment-status { payment_status: 'paid' }`
5. Статус → `paid`
6. Система логирует действие в `admin_audit_log`

### 5. Выполнение работ

1. Админ назначает исполнителя (`executor_id`) → статус → `in_progress`
2. Исполнитель выполняет работу
3. Обе стороны подтверждают через PWA (`POST /api/orders/:id/confirm`):
   - Заказчик: по ссылке из dashboard
   - Исполнитель: через `/order/:id/confirm?role=supplier`
4. Статус → `confirming`

### 6. Выплата исполнителю

1. Админ открывает заказ → вкладка «Выплата»
2. Выбирает способ выплаты (СБП / наличные)
3. Если СБП — запрашивает реквизиты у исполнителя (из `workers.payout_card`)
4. Выполняет перевод вручную через банковское приложение
5. В админ-панели: «Подтвердить выплату»
6. Или через API: `PUT /api/admin/orders/:id/payment-status { payment_status: 'paid_to_executor' }`
7. Статус → `completed`

## Требования к документообороту

Для ИП/ООО заказчиков (при необходимости закрывающих документов):
- Выставляется счёт через админ-панель → `/api/admin/orders/:id/send-invoice`
- Акт выполненных работ генерируется через `/api/admin/documents`
- Закрывающие документы отправляются через MAX (не email!)

---

## Безопасность

- Все действия админа логируются в `admin_audit_log`
- PIN обязателен для всех админских операций
- Публичные API не возвращают `base_price`, `markup_percent`, `supplier_payout`
- Заказчик видит только `display_price`
- Админ видит все финансовые поля

## FAQ

**Q: Что если заказчик не платит после отправки счёта?**
A: Через 48 часов админ может отменить заказ (`cancelled`) или отправить напоминание.

**Q: Что если исполнитель не выходит на связь?**
A: Админ переназначает исполнителя (PUT /api/admin/orders/:id с новым executor_id).

**Q: Что если спор (dispute)?**
A: Админ выступает арбитром:
1. Открыть спор: POST `/api/orders/:id/dispute`
2. Решить: PATCH `/api/orders/:id/dispute` → `completed` или `cancelled`
3. При отмене — возврат денег заказчику (вручную через банк)

**Q: Можно ли подключить онлайн-оплату?**
A: Скаффолд есть в `/api/payment/create` (Tinkoff + YooKassa). Нужно:
1. Получить API-ключи в платёжном шлюзе
2. Прописать в `.env`: `PAYMENT_GATEWAY=tinkoff` + ключи
3. Настроить callback URL на Vercel: `https://podryad.pro/api/payment/callback`
4. Протестировать на тестовом шлюзе
